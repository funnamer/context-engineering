# **harness engineering**



在过去的几个月里,Claude 一直致力于解决两个相互关联的问题:让 Claude 产出高质量的前端设计,以及让它在无需人类干预的情况下构建完整的应用程序。这项工作源于我们早期在前端设计技能和长期运行的编码智能体框架方面所做的努力,在这些努力中,Claude 的同事和 Claude 能够通过提示词工程和框架设计将 Claude 的性能提升到远超基线的水平——但这二者最终都遇到了天花板。

为了突破瓶颈,Claude 寻找了在两个截然不同的领域(一个由主观品味定义,另一个由可验证的正确性和可用性定义)均适用的新型 AI 工程方法。从生成对抗网络(GANs)中汲取灵感,Claude 设计了一个包含"生成器(generator)"和"评估器(evaluator)"智能体的多智能体结构。构建一个能够可靠且有品味地对输出进行评分的评估器,意味着首先要开发一套标准,将"这个设计好吗?"等主观判断转化为具体的、可评分的条款。

然后,Claude 将这些技术应用于长期运行的自主编码,并延续了我们早期框架工作中的两个经验教训:将构建过程分解为易于处理的块(chunks),以及使用结构化的工件(artifacts)在会话之间传递上下文。最终的结果是一个三智能体架构——计划者(planner)、生成器(generator)和评估器(evaluator)——它们在长达数小时的自主编码会话中生成了丰富的全栈应用程序。

## **为什么朴素的实现方式会失败**
我们之前已经证明，框架设计对长期运行的智能体编码的有效性有着重大影响。在早期的一项实验中，我们使用了一个初始化智能体将产品规格分解为任务列表，并使用一个编码智能体一次实现一个功能的任务，然后在会话之间移交工件以传递上下文。更广泛的开发者社区也得出了类似的见解，例如使用钩子或脚本让智能体保持在持续迭代循环中的“Ralph Wiggum”方法。

但一些问题依然存在。在分解这个问题时，我们观察到了智能体在执行此类任务时常见的两种失败模式。

首先,随着上下文窗口的填满,模型在冗长的任务中往往会失去连贯性(参见我们关于上下文工程的帖子)。一些模型还会表现出"上下文焦虑(context anxiety)",即当它们接近它们认为的上下文限制时,就会开始过早地结束工作。上下文重置(context resets)——完全清除上下文窗口并启动一个全新的智能体,结合携带前一个智能体状态和后续步骤的结构化移交——解决了这两个问题。

这不同于压缩(compaction),压缩是在原地总结对话的早期部分,以便同一个智能体可以在缩短的历史记录上继续工作。虽然压缩保持了连续性,但它并没有给智能体一个全新的开始,这意味着上下文焦虑可能仍然存在。重置提供了一个全新的开始,代价是移交工件必须具有足够的状态,以便下一个智能体能干净利落地接手工作。在我们早期的测试中,Claude 发现 Claude Sonnet 4.5 表现出了非常强烈的上下文焦虑,以至于仅靠压缩不足以实现强大的长任务性能,因此上下文重置成为框架设计的必要条件。这解决了核心问题,但给每次框架运行增加了编排复杂性、token 开销和延迟。

我们之前尚未解决的第二个问题是自我评估。当被要求评估自己产生的工作时,智能体往往会自信地赞美这些工作——即使对于人类观察者来说,质量明显很平庸。这个问题在像设计这样的主观任务中尤为突出,因为在这些任务中,没有等同于可验证软件测试的二进制检查。一个布局感觉是精致的还是普通的,这是一个判断题,智能体在给自己的工作评分时可靠地偏向于正面。

然而,即使在确实具有可验证结果的任务上,智能体有时也会表现出糟糕的判断力,从而阻碍它们在完成任务时的表现。将执行工作的智能体与评判工作的智能体分离开来,被证明是解决这个问题的有力手段。这种分离本身并不能立即消除那种宽容;评估器仍然是一个倾向于对 LLM 生成的输出表现慷慨的 LLM。但是,Claude 发现将一个独立的评估器调整为具有怀疑态度,事实证明比让一个生成器对自己工作持批评态度要容易得多,一旦存在外部反馈,生成器就有了具体的迭代目标。

## **前端设计：让主观质量变得可评分**
我首先在前端设计上进行了实验，自我评估问题在那里最为明显。在没有任何干预的情况下，Claude 通常倾向于安全、可预测的布局，这些布局在技术上能起作用，但在视觉上平淡无奇。

两个见解塑造了我为前端设计构建的框架。首先，虽然美学不能完全简化为一个分数——而且个人品味总是各不相同——但可以通过编码设计原则和偏好的评分标准来改善它们。“这个设计美吗？”很难一致地回答，但“这是否遵循了我们关于优秀设计的原则？”为 Claude 提供了具体的评分依据。其次，通过将前端生成与前端评分分离，我们可以创建一个反馈循环，推动生成器产出更强大的输出。

考虑到这一点,Claude 编写了四个评分标准,并在提示词中提供给生成器和评估器智能体:

* **设计质量 (Design quality):** 设计是否感觉像一个连贯的整体,而不是各个部分的集合?这里的出色表现意味着颜色、排版、布局、图像和其他细节结合在一起,创造了独特的情绪和特征。
* **原创性 (Originality):** 是否有定制决策的证据,或者这是否是模板布局、库默认值和 AI 生成的模式?人类设计师应该能认出深思熟虑的创意选择。未修改的组件库元素——或 AI 生成的明显迹象,如白色卡片上的紫色渐变——在此项不合格。
* **工艺 (Craft):** 技术执行力:排版层次结构、间距一致性、色彩和谐度、对比度。这是一项能力检查,而不是创造力检查。大多数合理的实现默认都能在这里做得很好;失败意味着基本功的破坏。
* **功能性 (Functionality):** 独立于美学的可用性。用户能否理解界面的作用,找到主要操作,并在无需猜测的情况下完成任务?

Claude 强调设计质量和原创性胜过工艺和功能性。由于所需的熟练技术往往是模型自然具备的,Claude 在默认情况下在工艺和功能性方面已经得分很高。但在设计和原创性方面,Claude 经常产生充其量也只是平庸的输出。这些标准明确惩罚了高度通用的"AI 劣质 (slop)"模式,并通过对设计和原创性赋予更高的权重,推动模型在美学上承担更多风险。

Claude 使用带有详细分数细分的少样本(few-shot)示例对评估器进行了校准。这确保了评估者的判断与 Claude 的偏好一致,并减少了跨迭代的分数漂移。

Claude 在 Claude Agent SDK 上构建了这个循环,这使得编排变得简单明了。生成器智能体首先根据用户提示词创建 HTML/CSS/JS 前端。Claude 给评估器配备了 Playwright MCP,这让它在对每个标准进行评分并撰写详细评论之前,能够直接与实时页面交互。在实践中,评估器会自行导航页面,截图并仔细研究实现情况,然后才得出评估结果。该反馈随后流回生成器,作为下一次迭代的输入。Claude 每次生成运行 5 到 15 次迭代,每次迭代通常都会推动生成器朝着更独特的方向发展,以响应评估者的批评。因为评估器是主动导航页面而不是对静态截图进行评分,所以每个周期都要花费真实的挂钟时间(wall-clock time)。完整的运行可能会长达四个小时。Claude 还指示生成器在每次评估后做出战略决策:如果分数趋势良好,则改进当前方向;如果方法不起作用,则完全转向完全不同的美学。

在不同运行中,评估器的评估在迭代中有所改善,然后趋于平稳,但仍留有提升空间。有些生成是增量改进的。其他则在迭代之间发生了急剧的美学转变。

标准的措辞以 Claude 没有完全预料到的方式引导了生成器。包含"最好的设计是博物馆级的质量"这样的短语,将设计推向了特定的视觉趋同,这表明与标准相关的提示直接塑造了输出的特征。

虽然分数在迭代过程中总体上有所提高,但模式并不总是纯粹线性的。后期的实现在整体上往往更好,但 Claude 经常看到 Claude 更喜欢中间某次迭代而不是最后一次迭代的情况。实现复杂性在各个回合中也倾向于增加,生成器试图通过更雄心勃勃的解决方案来回应评估者的反馈。即使在第一次迭代中,输出也明显好于完全没有提示的基线,这表明这些标准和相关语言本身就将模型从通用默认值中引导出来,而这发生在一个评估者反馈导致进一步细化之前。

在一个显著的例子中,Claude 提示模型为一个荷兰艺术博物馆创建一个网站。到了第九次迭代,它已经为一个虚构的博物馆生成了一个干净的、暗色主题的登陆页面。该页面在视觉上很精致,但在很大程度上符合 Claude 的期望。然后,在第十个周期,它完全抛弃了这种方法,并将网站重新想象为一种空间体验:一个具有 CSS 透视渲染的方格地板的 3D 房间,艺术品以自由形式悬挂在墙上,并且画廊房间之间的导航基于门,而不是滚动或点击。这是 Claude 以前在单次生成中从未见过的创造性飞跃。

## **扩展到全栈编码**
掌握了这些发现后,Claude 将这种受 GAN 启发的模式应用于全栈开发。生成器-评估器循环自然地映射到软件开发生命周期中,在这个周期中,代码审查和 QA 扮演着与设计评估器相同的结构性角色。

## **架构**
在我们早期的长期运行框架中,我们通过一个初始化智能体、一个一次处理一个功能的编码智能体以及会话之间的上下文重置,解决了连贯的多会话编码问题。上下文重置是一个关键的解锁:该框架使用了 Sonnet 4.5,它表现出了前面提到的"上下文焦虑"倾向。创建一个能在上下文重置中运行良好的框架,对于让模型保持专注至关重要。Opus 4.5 基本上自行消除了这种行为,所以 Claude 能够从这个框架中完全放弃上下文重置。这些智能体在整个构建过程中作为一个连续的会话运行,并由 Claude Agent SDK 的自动压缩功能处理沿途的上下文增长。

为这项工作,Claude 在原始框架的基础之上构建了一个三智能体系统,每个智能体解决 Claude 在之前的运行中观察到的一个特定差距。该系统包含以下智能体角色:

* **计划者 (Planner):** 我们之前的长期运行框架要求用户预先提供详细的规格说明。Claude 想将该步骤自动化,所以 Claude 创建了一个计划者智能体,它接受一个简单的 1-4 句提示词,并将其扩展为完整的产品规格。Claude 提示它在范围上要有野心,并专注于产品上下文和高层次的技术设计,而不是详细的技术实现。这种强调是因为担心如果计划者试图预先指定粗粒度的技术细节并且搞错了,规格说明中的错误将级联到下游的实现中。限制智能体要生产的交付物,让它们在工作过程中自行找出路径似乎更明智。Claude 还要求计划者寻找机会将 AI 功能融入到产品规范中。(请参阅底部附录中的示例。)
* **生成器 (Generator):** 早期框架中"一次一个功能"的方法在范围管理方面效果很好。Claude 在这里应用了类似的模型,指示生成器以冲刺(sprints)的方式工作,每次从规格中挑选一个功能。每个冲刺都使用 React、Vite、FastAPI 和 SQLite(后来是 PostgreSQL)技术栈实现应用程序,并且生成器被指示在每个冲刺结束时,在移交给 QA 之前对自己的工作进行自我评估。它还拥有用于版本控制的 git。
* **评估器 (Evaluator):** 早期框架生成的应用程序通常看起来令人印象深刻,但当您实际尝试使用它们时仍然存在真正的错误。为了捕捉这些错误,评估器使用 Playwright MCP 像用户一样点击正在运行的应用程序,测试 UI 功能、API 端点和数据库状态。然后,它根据发现的错误以及参照前端实验调整的、涵盖产品深度、功能性、视觉设计和代码质量的一套标准来对每个冲刺进行分级。每个标准都有一个硬性的阈值,如果任何一项低于该阈值,冲刺就会失败,生成器会收到关于哪里出了问题的详细反馈。

在每个冲刺之前,生成器和评估器会协商一份冲刺合同(sprint contract):在编写任何代码之前就该工作块的"完成"状态达成一致。它的存在是因为产品规范有意保持在高层次,并且 Claude 想要一个步骤来弥合用户故事和可测试的实现之间的差距。生成器提出它将构建什么以及如何验证成功,评估器则审查该提案以确保生成器正在构建正确的东西。两者反复迭代直到达成一致。

通信是通过文件处理的:一个智能体写入一个文件,另一个智能体读取该文件并在该文件内回复,或者使用上一个智能体接着会读取的新文件回复。然后,生成器根据商定的合同进行构建,然后将工作移交给 QA。这使得工作忠实于规格,而又不会过早地过度指定实现细节。

## **运行框架**
对于该框架的第一个版本,Claude 使用了 Claude Opus 4.5,将用户提示词在完整框架和单智能体系统上分别运行以进行比较。Claude 使用 Opus 4.5 是因为这是 Claude 开始这些实验时我们最好的编码模型。

Claude 写了以下提示词来生成一个复古视频游戏制作工具:
*“创建一个2D复古游戏制作工具，功能包括关卡编辑器、精灵编辑器、实体行为和可玩的测试模式。”*

下表显示了框架类型、运行时间和总成本。

| 框架 | 时长 | 成本 |
| :--- | :--- | :--- |
| 单次 (Solo) | 20 分钟 | $9 |
| 完整框架 (Full harness) | 6 小时 | $200 |

完整框架的成本高出 20 倍以上，但输出质量的差异是显而易见的。

Claude 期望得到一个界面,在其中可以构建一个关卡及其组成部分(精灵、实体、图块布局),然后点击"播放"实际玩这个关卡。Claude 首先打开了单次运行的输出,最初的应用程序似乎符合这些期望。

然而,当 Claude 点击浏览时,问题开始出现了。布局浪费了空间,固定高度的面版留下了大部分空白视图。工作流僵化。试图填充一个关卡时,它提示 Claude 先创建精灵和实体,但 UI 中没有任何东西引导 Claude 遵循该顺序。更重要的是,实际的游戏是坏的。Claude 的实体出现在屏幕上,但对输入没有任何响应。深入代码发现,实体定义与游戏运行时之间的连线断开了,表面上也没有任何迹象表明断在哪里。

在评估了单次运行之后,Claude 将注意力转向了框架运行。这次运行从同一句提示词开始,但计划者步骤将该提示词扩展为了一个分布在十个冲刺中的 16 个功能的规格说明。它远远超出了单次运行尝试的范围。除了核心编辑器和播放模式外,该规范还要求提供精灵动画系统、行为模板、音效和音乐、AI 辅助的精灵生成器和关卡设计师,以及带有可共享链接的游戏导出功能。Claude 让计划者可以访问我们的前端设计技能,它阅读并利用该技能为应用程序创建了视觉设计语言,作为规格说明的一部分。对于每个冲刺,生成器和评估器都协商了一份合同,定义了该冲刺的具体实现细节以及为验证完成而将进行测试的可测试行为。

应用程序立即展示出了比单次运行更多的精致和流畅。画布占据了整个视口,面版的大小合理,并且界面具有一致的视觉特征,符合规范中的设计方向。Claude 曾在单次运行中看到的一些笨拙之处确实保留了下来——工作流仍然没有明确表明在尝试填充关卡之前应该构建精灵和实体,Claude 必须通过四处摸索才能弄清楚。这被解读为基础模型产品直觉上的缺陷,而不是框架被设计来解决的问题,尽管这确实表明了框架内定向迭代可以帮助进一步提高输出质量的地方。

在使用编辑器时,新运行相对于单次运行的优势变得更加明显。精灵编辑器更加丰富,功能更加齐全,具有更简洁的工具调色板、更好的颜色选择器和更实用的缩放控件。

因为 Claude 曾要求计划者将 AI 功能编织到它的规范中,所以该应用程序还带有内置的 Claude 集成,允许 Claude 通过提示生成游戏的不同部分。这大大加快了工作流程。

（图片：开始屏幕、精灵编辑器、AI游戏设计、AI游戏设计、游戏玩法、初始屏幕：在新应用中创建新游戏）

最大的区别在于游玩模式。Claude 实际上能够移动 Claude 的实体并游玩游戏了。物理效果有些粗糙——Claude 的角色跳到一个平台上,结果却与平台重叠了,这在直觉上感觉不对——但核心部分生效了,而单次运行未能做到这一点。四处移动了一会儿之后,Claude 确实遇到了 AI 游戏关卡构建的一些限制。有一堵 Claude 无法跳过的高墙,所以 Claude 被卡住了。这表明有一些常识性的改进和边缘情况是框架可以处理以进一步完善应用程序的。

通读日志,可以清楚地看到评估器使实现保持符合规范。在每一个冲刺中,它都仔细检查了冲刺合同的测试标准,并通过 Playwright 运行应用程序,针对任何偏离预期行为的情况提交了错误(bugs)。合同颗粒度很细——仅在冲刺 3 中就有 27 个涵盖关卡编辑器的标准——评估器的发现足够具体,无需额外调查即可采取行动。下表显示了我们的评估器识别出的几个问题示例:

| 合同标准 | 评估器发现 |
| :--- | :--- |
| 矩形填充工具允许通过点击拖动，用选定的图块填充一个矩形区域 | **失败** — 工具仅在拖动起点/终点放置图块，而不是填充区域。`fillRectangle` 函数存在，但未能在 mouseUp 时正确触发。 |
| 用户可以选择并删除已放置的实体生成点 | **失败** — `LevelEditor.tsx:892` 处的删除键处理程序要求同时设置 `selection` 和 `selectedEntityId`，但点击实体仅设置了 `selectedEntityId`。条件应为 `selection || (selectedEntityId && activeLayer === 'entity')`。 |
| 用户可以通过 API 重新排序动画帧 | **失败** — `PUT /frames/reorder` 路由定义在 `/{frame_id}` 路由之后。FastAPI 将 'reorder' 匹配为 frame_id 整数并返回 422："无法将字符串解析为整数。" |

让评估器达到这种水平需要费一番功夫。开箱即用的情况下,Claude 是一个很差的 QA 智能体。在早期的运行中,Claude 看着它识别出了真正的问题,然后说服自己决定这些问题没什么大不了的,并无论如何都批准了这项工作。它还倾向于进行表面测试,而不是深入探究边缘情况,因此更微妙的错误经常溜走。调整循环就是读取评估器的日志,找到它的判断与 Claude 产生分歧的例子,并更新 QA 的提示词以解决这些问题。经过了几轮这样的开发循环后,评估器才以一种 Claude 认为合理的方式进行评分。即使在那时,框架的输出也显示了该模型 QA 能力的局限性:小的布局问题、在某些地方感觉不直观的交互,以及在更深层嵌套的特征中未被发现的错误(评估器没有彻底执行)。显然还有更多的验证提升空间可以通过进一步调优来捕获。但与单次运行(其中应用程序的中心功能根本不起作用)相比,提升是显而易见的。

## **迭代框架**
第一组框架结果令人鼓舞,但它也庞大、缓慢且昂贵。合乎逻辑的下一步是找到在不降低其性能的情况下简化框架的方法。这部分是常识,部分是由于一个更普遍的原则:框架中的每个组件都编码了一个关于模型靠自己做不到什么的假设,而这些假设值得进行压力测试,这既是因为它们可能不正确,也是因为随着模型的改进,它们可能很快就会过时。我们的一篇博客文章《构建有效的智能体》将其基本思想概括为"找到可能的最简单解决方案,并且只在必要时增加复杂性",这对于任何维护智能体框架的人来说都是一种一致呈现的模式。

在 Claude 第一次尝试简化时,Claude 大幅削减了框架并尝试了一些创新的想法,但 Claude 无法复制原始框架的性能。同时,也很难分辨出哪些框架设计的组成部分实际上是起到承载作用的,以及以何种方式起作用。基于这一经验,Claude 转向了更具条理的方法,一次移除一个组件,并审查它对最终结果的影响。

当 Claude 经历这些迭代周期时,我们也发布了 Opus 4.6,这为降低框架复杂性提供了进一步的动力。有充分的理由期望 4.6 需要的脚手架比 4.5 少。来自我们的发布博客:"[Opus 4.6] 计划更仔细,能够持续更长时间的智能体任务,在更大的代码库中运行更可靠,并且拥有更好的代码审查和调试技能来捕获其自身的错误。" 它还大幅改善了长上下文检索能力。这些正是该框架旨在补充的能力。

## **移除冲刺结构**
Claude 首先完全移除了冲刺(sprint)结构。冲刺结构曾帮助将工作分解为块,以便模型连贯地工作。鉴于 Opus 4.6 的改进,有充分的理由相信该模型可以原生处理该工作而无需这种分解。

Claude 保留了计划者和评估器,因为每一个都继续增加了明显的价值。如果没有计划者,生成器就会划定过小的范围:给定原始提示,它将在不首先对其工作进行规范化的情况下开始构建,最终创建一个功能不如计划者生成的应用程序丰富的应用程序。

随着冲刺结构的移除,Claude 将评估器移到了运行结束时的单次通过阶段,而不是按冲刺评分。因为模型变得更有能力,这改变了评估器在某些运行中承受负载的程度,其有用性取决于任务相对于模型独立可靠完成的界限。在 4.5 上,那个边界很近:我们的构建处于生成器单干能做好的边缘,评估器在整个构建过程中捕获了有意义的问题。在 4.6 上,模型的原始能力增强了,因此边界向外扩展了。过去需要评估器的检查才能连贯地实现的任务,现在通常在生成器单独能处理好的范围内了,对于该边界内的任务,评估器成了不必要的开销。但是,对于处于生成器能力边缘的构建部分,评估器继续提供实质性的提升。

实际的含义是,评估器并不是一个固定的非黑即白的抉择。当任务超出当前模型可靠的单干能力范围时,它是值得花这笔开销的。

除了结构简化之外,Claude 还添加了提示词来改进框架如何将 AI 功能构建到每个应用程序中,具体来说是让生成器构建一个合适的智能体,该智能体可以通过工具驱动应用程序本身的功能。这需要真正的迭代,因为相关知识足够新,以至于 Claude 的训练数据涵盖得很薄弱。但在经过足够的调优后,生成器能够正确构建智能体了。

## **更新后的框架结果**
为了测试更新后的框架,Claude 使用了以下提示词生成了一个数字音频工作站 (DAW),这是一个用于作曲、录音和混音的音乐制作程序:
*“使用 Web Audio API 在浏览器中构建一个全功能的 DAW。”*

运行过程仍然漫长且昂贵,花费了大约 4 个小时和 124 美元的 token 成本。

大部分时间花在了构建器上,它连贯运行了两个多小时,而无需 Opus 4.5 所需的冲刺分解。

| 智能体 & 阶段 | 时长 | 成本 |
| :--- | :--- | :--- |
| 计划者 (Planner) | 4.7 分钟 | $0.46 |
| 构建 (第1轮) | 2 小时 7 分钟 | $71.08 |
| QA (第1轮) | 8.8 分钟 | $3.24 |
| 构建 (第2轮) | 1 小时 2 分钟 | $36.89 |
| QA (第2轮) | 6.8 分钟 | $3.09 |
| 构建 (第3轮) | 10.9 分钟 | $5.88 |
| QA (第3轮) | 9.6 分钟 | $4.06 |
| **整体 V2 框架** | **3 小时 50 分钟** | **$124.70** |

与之前的框架一样,计划者将单行提示词扩展为了完整的规范说明。从日志中 Claude 可以看出,生成器模型在规划应用和智能体设计、连接智能体以及在移交给 QA 之前进行测试方面做得很好。

话虽如此,QA 智能体仍然发现了真正的漏洞。在它的首轮反馈中,它指出:
> 这是一个强大的应用程序，具有出色的设计保真度、坚实的 AI 智能体和良好的后端。主要的失败点是**特征完整性** — 虽然应用程序看起来令人印象深刻并且 AI 集成运行良好，但几个核心 DAW 功能只有显示而没有交互深度：剪辑无法在时间轴上拖动/移动，没有乐器 UI 面板（合成器旋钮、鼓垫），也没有视觉效果编辑器（EQ曲线、压缩器电平表）。这些不是边缘情况——它们是使 DAW 可用的核心交互，并且规范明确要求了它们。

在其第二轮反馈中，它再次抓住了几个功能上的差距：
> 剩余漏洞：
> - 音频录制仍仅为存根（按钮可以切换但无法捕获麦克风）
> - 未实现通过拖动边缘调整剪辑大小和剪辑分割
> - 效果可视化是数字滑块，不是图形（没有 EQ 曲线）

听之任之时,生成器仍然容易遗漏细节或仅留下功能的存根,而 QA 在发现这些"最后一公里"问题以供生成器修复方面依然增加了价值。

基于该提示词,Claude 期望得到一个程序,在其中 Claude 可以创作旋律、和声和鼓点模式,将它们编排成一首歌曲,并在过程中获得集成智能体的帮助。下面的视频显示了结果。
